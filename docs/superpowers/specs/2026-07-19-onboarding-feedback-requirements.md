# Hermes WebUI 上手体验与用户反馈闭环需求文档

**版本：** v1.0

**日期：** 2026-07-19

**状态：** 已完成工程评审，待线上验收

**优先级：** P0
**适用范围：** Neowow 线上部署、Hermes WebUI 桌面/本地部署

## 1. 背景

当前用户反馈集中在两类问题：

1. 首次使用时，不清楚登录、套餐同步、模型初始化分别处于什么状态，也不知道何时可以真正开始对话。
2. 登录、模型、配额或网络出现问题时，页面只有技术性错误提示，缺少可执行的恢复动作；“报告问题”入口又位于设置的日志页面，用户往往找不到。

现有实现同时存在独立的 Neowow 登录遮罩和 onboarding wizard，两边分别查询、维护状态。登录完成后，前端还会在 provider 激活请求尚未确认成功时显示“已就绪，正在启动”，并固定延时关闭遮罩。结果是用户可能进入一个看似完成、实际无法对话的状态。

另外，Neowow 登录授权有明确环境约束：**只有实际线上部署才能完成登录授权**。本地或桌面环境不能把 OAuth 登录展示为可完成的操作，必须说明限制并提供当前环境真正可用的路径。

## 2. 与现有设计的关系

本需求不重新设计已经确认的所有 onboarding 内容，而是统一其状态、反馈和恢复机制：

- 延续《3-Step Onboarding Redesign》的三步主流程：登录/连接能力 → 套餐 → 人格与启动。
- 延续《初始化“纯本地 Gemma”选项设计》的本地模型能力，仅在符合条件的本地桌面环境展示。
- 废止《First-Launch Login Onboarding Overlay》中“provider 激活请求发出后固定约 800ms 自动进入主页”的完成判定。
- “报告问题”功能继续保留，但从日志页工具扩展为全流程用户反馈入口。

如本文与旧登录遮罩方案冲突，以本文的状态机和完成条件为准。

## 3. 产品目标

### 3.1 核心目标

- 用户随时知道系统正在执行哪一步、是否成功、还需要等待多久或做什么。
- 只有在聊天能力经过服务端确认后，界面才声明“已就绪”。
- 登录、模型、配额和网络错误均提供可直接执行的恢复动作。
- 用户在登录前、首次引导中和聊天失败后都能快速报告问题。
- 首次用户以最短路径完成第一条有效对话，再逐步了解高级能力。

### 3.2 成功指标

上线后连续观察 14 天，目标如下：

| 指标 | 目标 |
|---|---:|
| 线上新用户 onboarding 完成率 | ≥ 80% |
| 登录开始到首条消息成功的中位耗时 | ≤ 3 分钟 |
| 登录成功但聊天未就绪的无提示失败率 | < 1% |
| 可恢复错误中，用户通过页面动作恢复成功的比例 | ≥ 50% |
| 问题报告提交成功率 | ≥ 95% |
| 首次使用阶段因“不知道下一步”产生的反馈量 | 相比上线前下降 50% |

如果当前没有基线数据，发布前至少连续采集 7 天基线。

## 4. 非目标

- 不修改 Neowow Dashboard 的套餐、计费、试用资格和 OAuth 服务端规则。
- 不在本需求中新增模型供应商或修改模型价格。
- 不重做整个首页、任务、技能、记忆等高级模块。
- 不让本地环境模拟、绕过或伪造线上 OAuth 登录授权。
- 不在问题报告中上传完整会话内容或工作区文件。

## 5. 用户与使用场景

### 5.1 线上新用户

用户打开实际线上部署地址，登录 Neowow，系统同步套餐和模型，选择必要的首次配置，成功发送第一条消息。

### 5.2 线上老用户

用户已有有效登录和 provider 配置，打开页面后直接进入聊天；如果套餐或模型同步暂时失败，使用上一次成功缓存，并明确提示当前状态。

### 5.3 本地或桌面用户

用户在本地打开 WebUI。系统明确说明“Neowow 登录授权仅支持线上部署”，不得打开一个无法完成的 OAuth 流程。页面根据环境能力展示可用路径，例如自有 API Key 或本地模型；若当前发行版不开放这些路径，则提供部署说明和退出/稍后处理入口。

### 5.4 遇到错误的用户

用户可能遇到登录过期、套餐同步超时、模型不存在、余额/配额不足、网络失败或未知错误。页面应解释发生了什么，并给出一到两个最相关的恢复动作，而不是要求用户理解日志或执行终端命令。

## 6. 统一状态模型

### 6.1 原则

- 前端只渲染一个 onboarding 容器，不再由登录遮罩和 wizard 各自决定是否阻断页面。
- 服务端是 readiness 的唯一事实来源。
- “登录成功”不等于“聊天就绪”。
- 所有中间状态必须可重入，刷新页面后能够继续，而不是从头开始。
- 同一时刻只允许一个主阶段，避免两个遮罩或两个加载状态并行。

### 6.2 阶段定义

| 阶段 | 含义 | 用户主要动作 |
|---|---|---|
| `environment_checking` | 正在判断线上/本地能力 | 等待 |
| `auth_required` | 线上环境需要登录 | 登录 |
| `auth_pending` | 已发起授权，等待回调 | 继续等待、重新打开登录 |
| `auth_unavailable_local` | 当前环境不能完成线上授权 | 查看部署说明、选择本地可用路径 |
| `provider_syncing` | 正在同步套餐、模型并激活 provider | 等待、取消后重试 |
| `provider_error` | provider 激活或模型同步失败 | 重试、重新登录、报告问题 |
| `first_run_setup` | 完成必要的首次配置 | 继续 |
| `ready` | 服务端确认可以开始聊天 | 进入聊天 |
| `recoverable_error` | 已进入产品后的可恢复错误 | 执行对应恢复动作 |

### 6.3 建议状态接口

优先扩展现有 `/api/onboarding/status`，不新增第二套并行状态接口。返回结构至少包含：

```json
{
  "deployment_mode": "online",
  "stage": "provider_syncing",
  "authenticated": true,
  "provider_configured": false,
  "chat_ready": false,
  "progress": {
    "step": "models",
    "message": "正在同步套餐和模型",
    "started_at": "2026-07-19T10:00:00Z"
  },
  "available_actions": ["retry", "relogin", "report_issue"],
  "error": null
}
```

约束：

- 只有 `chat_ready=true` 时前端才能显示“已就绪”并关闭 onboarding。
- `deployment_mode` 至少区分 `online`、`local_desktop` 和 `local_server`。
- `available_actions` 由服务端根据错误类型返回，前端不通过错误字符串猜测动作。
- `error` 对象只返回可展示的错误代码、用户文案和脱敏后的技术摘要。

## 7. 功能需求

### FR-01 环境能力识别

系统必须在展示登录按钮前识别当前部署模式。

验收标准：

- 实际线上部署显示“登录 Neowow”主按钮。
- 本地/桌面环境不显示可点击但无法完成的 Neowow OAuth 主按钮。
- 本地页面明确显示：“Neowow 账号授权仅支持线上部署。”
- 本地页面提供“查看线上部署说明”；若当前构建支持 API Key 或本地模型，同时显示对应入口。
- 手工调用本地不支持的登录启动接口时，服务端返回明确的业务错误码，而不是静默失败。

### FR-02 单一 onboarding 容器

登录、套餐/模型同步、首次设置必须在同一个 onboarding 容器中完成。

验收标准：

- 首次打开页面时最多出现一个阻断式容器。
- 刷新页面后恢复到服务端记录的当前阶段。
- 不再出现 boot overlay 消失后又被第二个独立遮罩覆盖的闪烁。
- 老用户在 `chat_ready=true` 时不进入 onboarding。

### FR-03 登录与 provider 激活反馈

线上登录成功后，系统必须继续展示 provider 同步进度，直到聊天能力确认可用。

验收标准：

- 点击登录后立即显示“等待登录授权”，按钮进入明确的等待状态。
- 收到登录回调后显示“登录成功，正在同步套餐和模型”。
- provider 激活成功且 `chat_ready=true` 后才显示“已就绪”。
- provider 激活失败时 onboarding 保持打开，不得固定延时进入主页。
- 超过 5 秒时展示持续等待说明；超过服务端超时时间后进入可重试错误状态。
- 使用上次成功模型缓存时明确提示“实时同步失败，当前使用上次成功配置”，但不阻断聊天。

### FR-04 首次成功路径

首次用户的主路径必须围绕“完成第一条有效对话”，高级能力在成功后再介绍。

主路径：

1. 连接可用的账号或模型能力。
2. 确认套餐和模型已就绪。
3. 完成必要的首次设置；可跳过的内容不得阻断。
4. 进入一个带示例输入的空白聊天。
5. 首条消息成功后标记 onboarding 完成。

验收标准：

- 首条消息成功前，不主动弹出技能、记忆、任务、消息渠道等高级模块介绍。
- 首页显示不超过 3 个与首次对话直接相关的建议。
- 首条消息成功后，可通过非阻断式清单介绍工作区、技能和自动任务。

### FR-05 结构化错误与恢复动作

错误必须按业务类型映射为用户可理解的说明和恢复动作。

| 错误类型 | 用户说明 | 主动作 | 次动作 |
|---|---|---|---|
| `auth_expired` | 登录已过期 | 重新登录 | 报告问题 |
| `auth_failed` | 登录未完成或授权失败 | 重新登录 | 查看详情 |
| `plan_sync_failed` | 套餐信息暂时无法同步 | 重试 | 使用上次配置 |
| `model_unavailable` | 当前模型暂不可用 | 切换模型 | 重试 |
| `quota_exceeded` | 当前套餐额度不足 | 查看套餐 | 切换可用模型 |
| `network_error` | 网络连接失败 | 重试 | 查看连接说明 |
| `unknown_error` | 暂时无法完成操作 | 报告问题 | 查看技术详情 |

验收标准：

- 中文为默认用户文案；技术堆栈默认折叠。
- 用户无需执行终端命令即可完成常见恢复操作。
- 每个错误最多突出两个动作，避免按钮堆叠。
- 重试操作必须防止连续重复提交，并展示进行中状态。

### FR-06 全流程“报告问题”入口

问题报告入口必须出现在问题发生处，而不仅位于日志页。

必须覆盖：

- 登录等待或登录失败页面。
- provider/模型同步失败页面。
- 聊天错误消息卡片。
- 全局未知错误提示。
- 设置 → 日志页。

验收标准：

- 用户最多点击一次即可从错误提示打开报告弹窗。
- 弹窗自动携带脱敏后的错误代码、当前阶段和关联请求 ID。
- 登录前也能创建报告；如果上传要求身份认证，应支持生成本地报告包并明确告知保存位置。
- 提交失败时保留已填写说明，支持重试或保存到本地。

### FR-07 报告预览、同意与脱敏

报告发送前必须让用户知道将上传什么。

验收标准：

- 展示拟上传文件清单、总大小、时间范围和脱敏说明。
- 用户可填写问题描述，并可取消任一非必要附件。
- 用户确认后才开始上传，不得打开弹窗即上传。
- 文案使用：“将自动识别并脱敏常见敏感信息，请提交前确认。”不得承诺绝对脱敏。
- 至少覆盖 API Key、JWT/Bearer、Cookie、常见云厂商 token、邮箱、用户目录、URL 查询参数中的敏感字段。
- 默认不上传完整用户对话、工作区文件内容和未脱敏环境变量。
- 离线待上传目录必须遵循当前激活的 `HERMES_HOME`/profile，而不是固定写入系统用户目录。

### FR-08 进度与性能反馈

任何超过 300ms 的关键操作都必须显示即时反馈；超过 5 秒时显示阶段说明。

验收标准：

- 登录等待、套餐同步、模型加载和报告上传均有独立进度文案。
- 同一页面不得重复请求相同 onboarding 状态；并发请求应合并或复用。
- 状态轮询使用退避策略，页面隐藏时降低轮询频率。
- 用户可以看到当前正在执行的业务步骤，但不显示内部实现细节。

### FR-09 可访问性与国际化

验收标准：

- onboarding 和问题报告弹窗使用共享 Dialog 结构及设计 token。
- 支持 `role="dialog"`、`aria-modal`、可感知标题、焦点进入/返回、焦点陷阱和 Escape 关闭；必须阻断的 onboarding 可禁用 Escape，但需提供原因。
- 所有主要操作可通过键盘完成。
- 中文和英文文案必须齐全，其他语言允许暂时回退英文。
- 加载、成功和错误不能只依赖颜色表达。

### FR-10 用户反馈漏斗数据

必须采集不包含用户内容和凭证的产品事件：

| 事件 | 必要字段 |
|---|---|
| `onboarding_view` | deployment_mode, stage |
| `auth_start` | deployment_mode |
| `auth_result` | result, error_code, duration_ms |
| `provider_activation_result` | result, error_code, duration_ms, used_cache |
| `first_message_result` | result, error_code, duration_from_onboarding_ms |
| `recovery_action_click` | error_code, action |
| `report_open` | source, error_code |
| `report_result` | result, error_code, bundle_size_bucket |

约束：

- 不记录用户输入、回复内容、JWT、API Key、完整文件路径或工作区名称。
- 用户拒绝产品分析数据时，不发送非必要埋点；本地部署默认遵循现有遥测开关。
- request ID 仅用于短期关联，不得作为跨产品永久用户标识。

### FR-11 遗留入口与文档一致性

验收标准：

- 确认根目录旧安装器是否仍属于发布路径。
- 非发布路径必须标注为 legacy，避免维护者误改。
- README、桌面打包入口、线上部署文档和 onboarding 行为保持一致。
- 文档明确区分“线上账号授权”和“本地模型/API Key 配置”。

## 8. 页面与交互要求

### 8.1 线上登录页

- 标题：连接 Neowow 账号
- 说明：登录后将自动同步可用套餐和模型。
- 主按钮：登录 / 注册 Neowow
- 辅助说明：登录将在安全的新窗口中完成。
- 辅助入口：遇到问题、查看隐私说明。

### 8.2 本地环境说明页

- 标题：当前为本地运行环境
- 说明：Neowow 账号授权仅支持实际线上部署，本地环境无法完成此登录。
- 主按钮：查看线上部署说明
- 条件入口：使用自己的 API Key、使用本地模型。
- 不得使用“正在验证”“稍后自动完成”等会造成错误预期的文案。

### 8.3 同步进度页

- 阶段依次显示：验证账号 → 获取套餐 → 同步模型 → 激活聊天能力。
- 已完成阶段显示明确完成标识。
- 当前阶段显示进行中状态和耗时说明。
- 失败时保留已经成功的阶段，不让用户从头重复登录。

### 8.4 聊天错误卡片

- 第一行说明用户影响，例如“当前模型暂时无法使用”。
- 第二行说明建议，例如“可以切换到 Kimi K3 后重试”。
- 操作区最多两个主要按钮。
- “技术详情”折叠区展示错误码、request ID 和时间，不展示敏感信息。

## 9. API 与数据要求

- `/api/onboarding/status`：统一返回部署环境、当前阶段、readiness、进度、错误和可用动作。
- `/api/neowow/activate-provider`：必须等待套餐/模型同步和配置写入完成后返回；响应明确包含 `chat_ready`。
- `/api/report-issue`：接收用户确认后的报告包；返回可展示的报告编号。
- 所有失败响应使用稳定的业务错误码，避免前端解析英文错误字符串。
- 状态和错误接口不得返回 JWT、API Key、Cookie、完整 provider 响应或内部堆栈。

## 10. 发布范围

### P0：解决“无法判断是否可用”

- 单一 onboarding 状态机和容器。
- 线上/本地环境识别。
- `chat_ready` 服务端完成判定。
- 登录、provider 和聊天错误的结构化恢复动作。
- 登录前、同步失败和聊天错误中的“报告问题”。
- 报告预览、确认和核心敏感信息脱敏。
- 核心漏斗事件。

### P1：降低首次使用认知负担

- 首次成功路径和首条消息成功清单。
- 高级功能渐进展示。
- 完整键盘与辅助技术支持。
- 中英文文案和设计系统收敛。

### P2：维护性治理

- 清理或归档遗留安装器。
- 状态请求去重与性能优化。
- 扩充多语言翻译和长期漏斗看板。

## 11. 测试与验收计划

### 11.1 自动化测试

- 状态转换：每个阶段的合法前进、失败、重试和刷新恢复。
- 环境矩阵：online、local_desktop、local_server。
- provider 激活：成功、超时、无模型、使用缓存、缓存为空。
- 错误码到用户动作的映射。
- 问题报告脱敏、预览、取消附件、上传失败和离线保存。
- 弹窗键盘、焦点和 ARIA 属性。
- 埋点字段白名单，证明不会发送用户内容和凭证。

### 11.2 必须在线上部署验证的场景

以下场景不能用本地模拟作为最终验收：

1. 首次 OAuth 登录完整回调。
2. 登录后套餐和模型同步。
3. provider 激活并完成第一条真实消息。
4. 登录过期后的重新授权。
5. 线上问题报告上传与报告编号返回。

必须准备独立的线上 staging 地址和测试账号，OAuth 回调域名需要提前加入允许列表。

### 11.3 人工验收

- 桌面宽屏、窄屏和移动端布局。
- 中文默认语言及英文回退。
- 慢网、断网、接口 5xx、接口超时。
- 刷新、关闭后重新打开、重复点击登录。
- 键盘全流程和屏幕阅读器基本朗读顺序。

## 12. 发布与回滚

- 统一 onboarding 通过服务端功能开关逐步开启。
- 先在 staging 完成真实 OAuth E2E，再向少量线上用户灰度。
- 灰度期间同时监控 `auth_result`、`provider_activation_result`、首条消息成功率和错误报告量。
- 保留旧流程的短期回滚开关，但新旧流程不得同时渲染。
- 如果 provider 激活失败率或首条消息失败率显著升高，立即回滚到旧流程并保留新埋点用于定位。

## 13. 依赖与风险

### 依赖

- Neowow OAuth 回调和线上域名配置。
- Dashboard `/api/me/plan` 的可用性与稳定错误格式。
- Hermes provider 配置写入和模型列表接口。
- 问题报告接收端、报告编号和存储策略。

### 主要风险

- 旧登录遮罩与新状态机同时运行，造成双重阻断。
- 将登录成功误当作 provider 可用，继续产生“假就绪”。
- 套餐接口慢时，统一状态接口被同步阻塞。
- 脱敏范围不足导致日志上传隐私风险。
- 本地和线上能力判断只依赖前端，接口仍可被错误调用。

所有环境限制和 readiness 判断必须同时由后端保证，前端只负责展示。

### 评审中发现并修复的问题

1. 登录前问题报告曾被全局鉴权提前拦截；现仅在线上 Neodomain 模式开放同源、受限的匿名入口。
2. 旧自动配置路径会绕过新的 readiness 状态机；现仅用于已完成安装的遗留配置修复。
3. 状态接口可能被套餐接口拖慢最长约 16.5 秒；现改为缓存优先，显式激活时才刷新网络。
4. 已存在配置会让显式 provider 激活卡在覆盖确认；激活接口现明确允许覆盖托管 provider 配置。
5. 两个函数内局部导入遮蔽全局符号，导致设置保存和 onboarding setup 返回 500；已移除遮蔽并通过集成测试。
6. 报告弹窗关闭事件会冒泡到全局 Esc 快捷键并误完成 onboarding；现捕获并停止传播，托管引导不可用 Esc 跳过。
7. `confirm_upload` 曾接受任意真值，字符串也可能被当作上传授权；现只接受 JSON 布尔值 `true`。
8. 离线报告使用秒级文件名且权限未收紧；现增加随机后缀并在 POSIX 上设置 `0600`。
